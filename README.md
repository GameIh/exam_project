# Веб-приложение для сдачи экзаменов

Итоговый Django-проект для проведения экзаменов в веб-интерфейсе. В приложении реализованы роли администратора, преподавателя и студента, прохождение экзамена с таймером, автоматическая проверка ответов, просмотр результатов и демонстрационное наполнение базы данных.

Проект можно запускать полностью в Docker: контейнер Django, PostgreSQL и pgAdmin поднимаются одной командой.

## Реализованный функционал

### Общая логика

- авторизация пользователей по email и паролю;
- разделение доступа по ролям: администратор, преподаватель, студент;
- автоматическая переадресация пользователя после входа в нужный раздел;
- хранение данных в PostgreSQL;
- часовой пояс проекта настроен под Благовещенск: `Asia/Yakutsk`;
- шаблоны Django отображают реальные данные из базы данных, а не статичную верстку;
- добавлены демо-данные для быстрой демонстрации проекта;
- проект контейнеризирован через Docker Compose.

### Администратор

Администратор работает через стандартную Django admin-панель по адресу:

```text
http://127.0.0.1:8000/admin/
```

В админ-панели можно управлять:

- пользователями и их ролями;
- предметами;
- записями студентов на предметы;
- экзаменами;
- вопросами;
- вариантами ответов;
- правильными ответами;
- попытками прохождения;
- ответами студентов.

Для удобства админка настроена с фильтрами, поиском, inline-формами и быстрыми действиями публикации/снятия экзаменов с публикации.

### Студент

Студентский маршрут включает:

- личный кабинет студента;
- список доступных экзаменов;
- запуск экзамена;
- прохождение вопросов;
- сохранение выбранных/введенных ответов;
- таймер экзамена;
- автоматическую отправку попытки по окончании времени;
- просмотр результата;
- просмотр статуса ответов: верно/неверно/без ответа.

Студент видит только те экзамены, которые:

- опубликованы;
- относятся к предметам, на которые он записан;
- доступны по дате и времени;
- еще не превышают лимит попыток.

### Преподаватель

Кабинет преподавателя доступен по адресу:

```text
http://127.0.0.1:8000/teacher/
```

Преподаватель видит только свои предметы и экзамены. В кабинете отображаются:

- создание и редактирование экзаменов;
- настройка длительности, периода доступности, лимита попыток и публикации;
- управление составом вопросов экзамена;
- собственный банк тестовых и текстовых вопросов;
- количество экзаменов;
- количество попыток студентов;
- средний результат;
- последние попытки;
- список экзаменов преподавателя;
- детализация по каждому экзамену;
- статистика по вопросам;
- список студентов и история результатов выбранного студента.

Страница детализации экзамена:

```text
http://127.0.0.1:8000/teacher/exams/<id>/
```

На ней преподаватель может посмотреть попытки студентов, процент выполнения, время прохождения и статистику правильных ответов по вопросам.

### Проверка ответов

В проекте реализована автоматическая проверка:

- вопросов с одним вариантом ответа;
- текстовых вопросов.

Для вопросов с выбором варианта ответ сравнивается с правильным вариантом, заданным в базе.

Для текстовых вопросов ответ нормализуется перед проверкой:

- приводится к нижнему регистру;
- буква `ё` заменяется на `е`;
- удаляются лишние знаки препинания;
- схлопываются лишние пробелы.

Это позволяет засчитывать близкие по написанию ответы, если они совпадают с одним из допустимых правильных вариантов.

### Таймер экзамена

Таймер работает на двух уровнях:

- в браузере студент видит обратный отсчет;
- на сервере проверяется фактическое время окончания попытки.

Даже если пользователь отключит JavaScript или попытается обойти интерфейс, сервер закроет просроченную попытку и отправит ее на проверку.

## Основные сущности базы данных

### `User`

Пользователь системы. Используется кастомная модель пользователя с ролями:

- `admin`;
- `teacher`;
- `student`.

Вход выполняется по email.

### `Subject`

Учебный предмет. Предмет привязан к преподавателю.

### `SubjectEnrollment`

Запись студента на предмет. По этой связи определяется, какие экзамены доступны студенту.

### `Exam`

Экзамен по предмету. Содержит:

- название;
- описание;
- дату и время начала;
- дату и время окончания;
- длительность;
- лимит попыток;
- статус публикации.

### `Question`

Вопрос по предмету. Поддерживаются типы:

- один вариант ответа;
- текстовый ответ.

### `ExamQuestion`

Связь экзамена и вопроса. Позволяет задавать порядок вопросов и количество баллов за конкретный вопрос.

### `QuestionOption`

Варианты ответа для вопроса с выбором.

### `CorrectChoiceKey`

Правильный вариант для вопроса с выбором.

### `CorrectTextKey`

Допустимый правильный текстовый ответ.

### `ExamAttempt`

Попытка прохождения экзамена студентом. Хранит статус, время начала, время отправки, набранные баллы и максимальный балл.

### `AttemptAnswer`

Ответ студента на конкретный вопрос в рамках попытки.

## Запуск через Docker

Перед запуском должен быть установлен Docker Desktop.

Перейдите в папку проекта:

```powershell

```

Запустите контейнеры:

```powershell

```

После запуска будут доступны:

| Сервис | Адрес |
| --- | --- |
| Django-приложение | `http://127.0.0.1:8000/` |

| Django admin | `http://127.0.0.1:8000/admin/` |

При старте web-контейнер:

1. ждет готовности PostgreSQL;
2. создает схему `exam`, если ее нет;
3. применяет миграции Django;
4. при `SEED_DEMO_DATA=true` наполняет базу демонстрационными данными;
5. запускает Django-сервер на `0.0.0.0:8000`.

## Демо-данные

Команда наполнения базы:

```powershell
docker compose exec web python manage.py seed_demo_data
```

Она создает:

- пользователей разных ролей;
- предметы;
- записи студента на предметы;
- несколько экзаменов;
- вопросы разных типов;
- варианты ответов;
- правильные ответы;
- готовую завершенную попытку;
- одну незавершенную попытку.

Важно: команда `seed_demo_data` пересоздает демонстрационные данные экзаменов и попыток. Это удобно во время разработки и защиты, потому что проект быстро возвращается к предсказуемому состоянию.

Если нужно сохранять данные между перезапусками web-контейнера, в `docker-compose.yaml` поменяйте:

```yaml
SEED_DEMO_DATA: "false"
```

## Проверка проекта

Перед демонстрацией можно выполнить:

```powershell
docker compose exec web python manage.py check
docker compose exec web python manage.py makemigrations --check --dry-run
docker compose exec web python manage.py test core
```

Что проверяют команды:

- `check` проверяет настройки Django;
- `makemigrations --check --dry-run` проверяет, что модели и миграции синхронизированы;
- `test core` запускает тесты основной логики приложения.

Покрытые тестами сценарии:

- автоматическая проверка ответов студента;
- закрытие просроченной попытки сервером;
- запрет запуска экзамена до даты начала;
- запрет запуска экзамена без вопросов;
- ограничение количества попыток;
- запрет просмотра чужого результата студентом;
- доступ преподавателя к своему кабинету;
- запрет преподавателю открывать чужой экзамен;
- запрет студенту открывать кабинет преподавателя.

## Локальный запуск без Docker

Основной способ запуска проекта - Docker. Локальный запуск нужен только для разработки, если хочется запускать Django напрямую на машине.

1. Создайте виртуальное окружение:

```powershell
py -m venv .venv
```

2. Активируйте его:

```powershell
.\.venv\Scripts\Activate.ps1
```

3. Установите зависимости:

```powershell
py -m pip install -r requirements.txt
```

4. Запустите PostgreSQL и pgAdmin из Docker:

```powershell
docker compose up -d postgres pgadmin
```

5. Для локального запуска Django укажите, что база находится на `localhost`. Это можно сделать через переменную окружения:

```powershell
$env:POSTGRES_HOST="localhost"
```

6. Примените миграции:

```powershell
py manage.py migrate
```

7. Заполните базу демо-данными:

```powershell
py manage.py seed_demo_data
```

8. Запустите сервер:

```powershell
py manage.py runserver
```

Приложение будет доступно по адресу:

```text
http://127.0.0.1:8000/
```

## Особенности реализации

- Для авторизации используется кастомная модель пользователя `core.User`.
- Доступ к страницам ограничен декораторами и проверками ролей.
- Данные студента и преподавателя фильтруются на уровне запросов к базе.
- Попытка экзамена создается при старте экзамена и переиспользуется, если студент уже начал экзамен, но не завершил его.
- Просроченные попытки закрываются серверной логикой.
- Результат считается после отправки попытки или после истечения времени.
- Максимальный балл считается по вопросам, включенным в конкретный экзамен.
- Демо-данные можно безопасно пересоздавать во время разработки.
